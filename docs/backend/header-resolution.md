# Header-Based Tenant Resolution

> LankaCommerce Cloud - Header-Based Resolution Documentation
> SubPhase-06, Group-D (Tasks 43-54)

---

## Overview

LankaCommerce Cloud supports header-based tenant resolution for API
traffic. Clients (mobile apps, third-party integrations, webhooks)
identify the target tenant by sending an HTTP header instead of
relying on hostname-based resolution.

Header resolution complements the SubdomainResolver (Group-B) and
CustomDomainResolver (Group-C). It is restricted to API paths only.

---

## API-Only Scope (Task 43)

Header-based resolution is active only on URL paths that match the
TENANT_HEADER_PATHS setting:

- /api/ - REST API endpoints
- /mobile/ - Mobile app API endpoints
- /webhook/ - Incoming webhook receivers

Requests to other paths (HTML pages, admin, static assets) are
ignored by the HeaderResolver and fall through to the subdomain or
custom domain resolver.

This restriction prevents header-based resolution from interfering
with browser-based navigation, where the Host header determines the
tenant.

---

## Tenant Header Names (Task 44)

Two HTTP headers are supported:

### X-Tenant-ID (Primary)

The primary header carrying the tenant identifier (UUID or integer PK).

Client usage:

- Header name: X-Tenant-ID
- Header value: The tenant's primary key or UUID
- Example: X-Tenant-ID: 42
- Example: X-Tenant-ID: a1b2c3d4-e5f6-7890-abcd-ef1234567890

### X-Tenant-Slug (Fallback)

An alternative header carrying the tenant's URL slug (schema_name).

Client usage:

- Header name: X-Tenant-Slug
- Header value: The tenant's schema name (slug)
- Example: X-Tenant-Slug: acme-corp

### Precedence

When both headers are present, X-Tenant-ID takes precedence.
If X-Tenant-ID is missing or empty, the resolver falls back to
X-Tenant-Slug.

### Missing Header Handling

When neither header is present, the resolver returns None and falls
through to the next resolution strategy (subdomain or custom domain).
No error is raised for missing headers.

## Settings Configuration (Tasks 45, 51)

Header resolution is configured via Django settings in
config/settings/base.py.

### Settings Reference

- TENANT_HEADER_NAME: Primary header name.
  Default: "X-Tenant-ID"
  Overridable via environment variable: TENANT_HEADER_NAME

- TENANT_SLUG_HEADER: Slug header name.
  Default: "X-Tenant-Slug"
  Overridable via environment variable: TENANT_SLUG_HEADER

- TENANT_HEADER_PATHS: URL path prefixes where header resolution is active.
  Default: ["/api/", "/mobile/", "/webhook/"]
  Defined as a Python list in settings (not env-overridable by default).

- TENANT_DOMAIN_CACHE_TTL: Cache TTL for tenant lookups (shared with
  subdomain and custom domain resolvers).
  Default: 300 (5 minutes)

### Settings Location

All settings are defined in:
config/settings/base.py

Environment-specific overrides can be placed in:
config/settings/local.py (development)
config/settings/staging.py (staging)
config/settings/production.py (production)

### Allowed Paths Configuration (Task 51)

The allowed path prefixes control where header-based resolution
is active. To add a new path prefix:

1. Open config/settings/base.py (or the environment-specific file).
2. Add the path prefix to the TENANT_HEADER_PATHS list.
3. Restart the application server to apply the change.

Paths must include a trailing slash. Only prefix matching is used
(startswith), so /api/ matches /api/v1/products/ and all other
/api/ sub-paths.

Security consideration: Do not add paths that serve HTML pages or
admin interfaces. Header-based resolution is intended for API traffic
only. Browser-based requests should use subdomain or custom domain
resolution where the hostname is validated by the server.

## Header Extraction (Task 46)

The HeaderResolver reads tenant identifiers from the Django request
object using the request.META dictionary.

### Extraction Process

1. Django converts HTTP headers to META keys with HTTP\_ prefix,
   uppercase, and hyphens replaced by underscores.
   X-Tenant-ID becomes HTTP_X_TENANT_ID.
   X-Tenant-Slug becomes HTTP_X_TENANT_SLUG.

2. The resolver reads the primary header (X-Tenant-ID) first.

3. If not found, reads the fallback header (X-Tenant-Slug).

4. Header values are stripped of leading/trailing whitespace.

5. Empty strings after stripping are treated as missing.

### Missing Header Behaviour

When no tenant header is found:

- extract_header() returns (None, "none").
- The resolve() method returns None.
- No error response is generated by the resolver.
- The middleware can fall through to subdomain/custom domain resolution.

## Tenant Lookup and Caching (Tasks 47, 52)

After extracting the identifier, the resolver looks up the Tenant
model instance. Lookups are cached for performance.

### Lookup Strategy

By ID (source="id"):

- Attempts Tenant.objects.get(pk=identifier).
- Handles integer and UUID primary keys.
- Falls through to slug lookup if ID lookup fails.

By Slug (source="slug"):

- Attempts Tenant.objects.get(schema_name=identifier.lower()).
- The schema_name is the unique slug for the tenant's PostgreSQL schema.

### Not-Found Handling

When no tenant matches the identifier:

- lookup_tenant() returns None.
- The miss-sentinel is cached to prevent repeated DB queries.
- A debug log entry is emitted.

### Cache Behaviour (Task 52)

Tenant lookups are cached using Django's cache framework to reduce
database load and improve response times.

Cache configuration:

- Cache backend: Django's default cache (CACHES["default"]).
- Cache key pattern: lcc:tenant_header:{source}:{identifier}
- Cache TTL: TENANT_DOMAIN_CACHE_TTL (default 300 seconds / 5 minutes).
- Miss sentinel: "**none**" stored for not-found identifiers.

Cache invalidation:

- invalidate_header_cache(identifier): Module-level function that
  evicts the cache entry for a specific tenant identifier.
- Should be called from Tenant model post_save/post_delete signals.
- Both ID and slug cache entries should be cleared when a tenant
  is modified or deleted.

Cache sizing:

- Each cached tenant object occupies a small amount of memory.
- Miss-sentinel entries are lightweight (short string).
- For a system with 1000 tenants, the cache uses minimal memory.

Cache monitoring:

- Cache hits/misses are logged at DEBUG level.
- Monitor cache hit rate to ensure TTL is appropriate.
- If tenants are frequently modified, consider a shorter TTL.

## Tenant Validation (Task 48)

After lookup, the resolver validates the tenant before returning it.

### Validation Checks

1. Tenant is not None (exists in the database).
2. Tenant is active (is_active=True, if the field exists).

### Inactive Tenant Handling

When a tenant is found but is inactive:

- validate_tenant() returns False.
- A warning is logged with the tenant name.
- The resolve() method returns None.
- The caller should return HTTP 404 or HTTP 403.

### Error Responses

The HeaderResolver does not generate HTTP responses directly. It
returns None on failure. The calling middleware or view is responsible
for generating the appropriate error response:

- Missing header: Fall through to next resolver (no error).
- Tenant not found: HTTP 404.
- Tenant inactive: HTTP 403 or HTTP 404.

## API Authentication Integration (Task 49)

Header-based tenant identification is NOT authentication. The header
only selects which tenant schema to use. Authentication and
authorisation are separate concerns that must be handled independently.

### Security Boundary

The tenant header is a routing hint. It tells the middleware which
PostgreSQL schema to activate for the request. However, the caller
must still be authenticated and authorised to access that tenant's data.

### Authentication Flow

1. Client sends request with tenant header (X-Tenant-ID or X-Tenant-Slug).
2. HeaderResolver resolves the target tenant (routing step).
3. Authentication middleware verifies user identity (JWT, API key, session).
4. validate_user_tenant_access() checks user belongs to tenant.
5. DRF permission classes enforce object-level permissions.

### User-Tenant Access Validation

The HeaderResolver provides validate_user_tenant_access(request, tenant)
to check that an authenticated user belongs to the resolved tenant.
This prevents cross-tenant data access where a user changes the header
to target a tenant they do not belong to.

Behaviour:

- Anonymous requests: Returns True (authentication middleware handles it).
- Authenticated users with tenant membership: Returns True.
- Authenticated users without tenant membership: Returns False, logs warning.
- Users without a tenants relation: Returns True (permissive fallback).

### Security Recommendations

- Always use HTTPS to prevent header interception.
- Validate tenant headers in combination with authentication tokens.
- Do not trust tenant headers from untrusted sources without auth.
- Monitor failed tenant access attempts in audit logs.
- Consider rate limiting header-based resolution to prevent enumeration.

## Path Restriction Enforcement (Task 50)

Header-based resolution is enforced only on approved API routes.
Requests outside the allowed paths are rejected.

### Enforcement Behaviour

When a request arrives with a tenant header on a non-API path:

- is_header_path() returns False.
- The resolver logs a debug message with the rejected path.
- The resolve() method returns None.
- An audit log entry is created with outcome "path_rejected".
- The middleware falls through to subdomain or custom domain resolution.

### Why Restrict Paths

- HTML pages use hostname-based tenant resolution (subdomain/custom domain).
- Header-based resolution could confuse server-side rendered pages.
- Admin interfaces should not accept header-based tenant switching.
- Static assets and media files do not need tenant resolution.

### Adding New Paths

To add a new path prefix for header-based resolution:

1. Add the prefix to TENANT_HEADER_PATHS in Django settings.
2. Ensure the path serves API responses (JSON), not HTML.
3. Test that the path correctly resolves tenants via headers.
4. Document the new path in this file.

## Audit Logging (Task 53)

Header-based tenant access is logged for auditing and security monitoring.

### What Is Logged

Every header-based tenant resolution produces a log entry containing:

- outcome: The resolution result (resolved, rejected, not_found, inactive, path_rejected).
- tenant: The tenant name or identifier (or "none" for failures).
- path: The request path.
- method: The HTTP method (GET, POST, PUT, DELETE, etc.).
- user: The authenticated username (or "anonymous").

### Log Levels

- INFO: Successful tenant resolution (outcome="resolved").
- WARNING: Failed resolution (rejected, not_found, inactive, path_rejected).

### Log Format

The log message format is:

    HeaderResolver audit: outcome={outcome} tenant={tenant} path={path} method={method} user={user}

### Audit Trail Requirements

- Logs should be forwarded to a centralised logging system (ELK, CloudWatch, Datadog).
- Retain audit logs for at least 90 days for compliance.
- Monitor WARNING-level entries for potential security incidents.
- Alert on repeated "rejected" outcomes from the same user or IP.
- Use structured logging (JSON) in production for easier parsing.

### Integration with Django Logging

The HeaderResolver uses the standard Python logging module with the
logger name "apps.tenants.middleware.header_resolver". Configure this
logger in Django's LOGGING setting to control output format and handlers.

## HeaderResolver Class

The HeaderResolver class lives in:

    apps.tenants.middleware.header_resolver

### Key Methods

- is_header_path(path): Check if path is eligible for header resolution.
  Enforces path restriction with rejection logging (Task 50).
- extract_header(request): Read tenant identifier from request headers.
- lookup_tenant(identifier, source): Cached tenant lookup by ID or slug (Task 52).
- validate_tenant(tenant): Check tenant exists and is active.
- validate_user_tenant_access(request, tenant): Check authenticated user
  belongs to resolved tenant (Task 49).
- log_header_access(request, tenant, outcome): Audit log entry (Task 53).
- resolve(request): Combined extract + lookup + validate + auth + log.

### Module-Level Functions

- invalidate_header_cache(identifier): Evict cached header lookup.

## Resolution Flow (Task 54)

The complete header-based tenant resolution flow:

1. HTTP request arrives at an API endpoint (e.g. /api/v1/products/).
2. LCCTenantMiddleware (or custom middleware) invokes HeaderResolver.resolve(request).
3. HeaderResolver.is_header_path("/api/v1/products/") checks path eligibility.
   - If path is NOT in allowed list: log rejection, return None.
4. HeaderResolver.extract_header(request) reads X-Tenant-ID or X-Tenant-Slug.
   - If no header found: return None (fall through to other resolvers).
5. HeaderResolver.lookup_tenant(identifier, source) queries cache, then DB.
   - Cache hit: return cached tenant immediately.
   - Cache miss: query DB, cache result (or miss sentinel).
6. HeaderResolver.validate_tenant(tenant) confirms tenant is active.
   - If tenant is None: log not_found, return None.
   - If tenant is inactive: log inactive, return None.
7. HeaderResolver.validate_user_tenant_access(request, tenant) checks auth.
   - Anonymous: pass through (auth middleware handles it).
   - Authenticated without membership: log rejected, return None.
8. HeaderResolver.log_header_access(request, tenant, "resolved") creates audit entry.
9. Tenant schema is activated for the request.
10. Request proceeds to the API view.

### Constraints

- Header is NOT authentication — always use with auth middleware.
- Only active on allowed paths — non-API paths are rejected.
- Cache TTL is configurable — tune for your tenant change frequency.
- All outcomes are audit-logged — monitor for security incidents.
- Inactive tenants are rejected — no silent passthrough.
- Cross-tenant access is blocked — user must belong to tenant.

---

## Integration with Other Resolvers

The three resolvers form a resolution chain:

1. HeaderResolver (API paths only): Checks for tenant header.
2. SubdomainResolver: Extracts subdomain from Host header.
3. CustomDomainResolver: Matches full custom domain.

The middleware tries each resolver in order. The first successful
resolution wins. If all return None, the request falls through to
the public schema or returns HTTP 404.
